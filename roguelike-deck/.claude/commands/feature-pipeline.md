# ゲーム機能追加パイプライン

2エージェント + Codex パターン（計画 → Codex 実装 → 評価）でゲーム機能を追加します。

## 手順

1. **planner** エージェントに `$ARGUMENTS` を渡し、実装計画を作成させる
   - 計画には「Codex 実装タスク」テーブルが含まれること（全ファイルを対象にする）
2. 計画の内容（特にスプリント契約・タスク分割）をユーザーに確認する（承認されるまで進めない）
3. **Codex CLI** で全タスクをシリアル実装する（TDD で実行される）：
   - 計画の「Codex 実装タスク」テーブルを読み取り、タスクを1件ずつ順番に実行する
   - 各タスクのプロンプトは「計画ファイルの内容 + タスクID + 担当ファイル + 完了条件」を渡す
   - 実行コマンド例：
     ```bash
     npx @openai/codex exec --sandbox workspace-write \
       "計画: $(cat .claude/plans/<plan-file>.md)
        タスク: T1
        担当ファイル: src/core/battle.ts
        完了条件: ダメージ・ブロック計算のテストがGREEN"
     ```
   - **重要**: Bash ツールで実行する際は必ず `run_in_background: false`（デフォルト）を使うこと。`run_in_background: true` は絶対に使わない。ユーザーがリアルタイムで進捗を確認できることが必須条件
   - 各タスク完了後、次のタスクへ進む（シリアル実行）
   - ※並列実行（`&` + `wait`）は未検証。現時点ではシリアルのみ
4. **evaluator** エージェントに計画ファイルのパスと `git diff HEAD` を渡し、レビューさせる
5. evaluator の判定に応じて：
   - **PASS**：`git add` + `git commit` をまとめて行い、完了レポートを出力して終了
   - **NEEDS_REVISION**：指摘された該当タスクを `codex exec` で再実行する（最大3回）
     - 複数ファイルにまたがる統合の修正が必要な場合のみ、Claude Code（メインセッション）が直接修正する
   - **FAIL（3回連続）**：planner からやり直すことをユーザーに提案する
6. 3イテレーション以内に収束しない場合はユーザーに判断を仰ぐ

## 引数

`$ARGUMENTS` を実装したいゲーム機能の要件として使用する。

## 使用例

```
/feature-pipeline 均衡型・連撃型の開始デッキ選択画面と初期遺物効果を実装する
/feature-pipeline オリジナルカード工房で素材カード2枚を選んで合成し、コスト計算と代償付与を行う
/feature-pipeline 敵図鑑ポイントの加算処理と、満タン時にエネミーオーブを解放する仕組みを実装する
/feature-pipeline 鍛冶ノードでカード鍛冶（空きスロットへ効果付与）と遺物変換のどちらかを選べるようにする
/feature-pipeline 進化カードの使用回数カウントと進化条件達成時のカード置き換え処理を実装する
/feature-pipeline ポーション所持枠（上限2）と戦闘中のポーション使用処理を実装する
```

## 注意事項

- Renderer分離の原則を絶対に守ること（plannerに確認させる）
- オリジナルカードの複製禁止・エネミースロットの鍛冶禁止など、仕様固有の制約を planner に意識させること
- ゲームバランスの数値は仮置きでOK。後でバランス調整スプリントで修正する
- Codex は TDD（RED→GREEN→REFACTOR）で実装するため、各タスクの完了条件に「テストがGREEN」を含めること
